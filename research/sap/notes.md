# SAP ERP Procurement Ingestion Research Notes

This document provides staff-level engineering research and onboarding analysis for extracting and ingesting fuel and materials procurement logs from enterprise SAP ERP instances.

---

## 1. Typical Export Structure

In enterprise contexts, fuel procurement data resides in **Materials Management (MM)** and **Financial Accounting/Controlling (FI/CO)** modules. Companies rarely establish live API endpoints for initial historical audits; instead, auditors extract flat ALV grid ledger files using standard transaction codes:

*   **Transaction `MB51` (Material Document List)**: Extracts physical inventory movements. Combines data from:
    *   **`MKPF`** (Material Document Header): Contains `MBLNR` (Doc Number), `BUDAT` (Posting Date), and `USNAM` (User).
    *   **`MSEG`** (Material Document Segment): Contains `WERKS` (Plant Code), `MATNR` (Material Number), `MENGE` (Quantity), `MEINS` (Base Unit of Measure), and `KOSTL` (Cost Center).
*   **Transaction `ME2N` (Purchase Orders by PO Number)**: Extracts purchase ledgers. Combines data from:
    *   **`EKKO`** (Purchasing Document Header): Contains `EBELN` (PO Number), `LIFNR` (Supplier Code), `WAERS` (Currency), and `BEDAT` (Doc Date).
    *   **`EKPO`** (Purchasing Document Item): Contains `MATNR` (Material Group/Code), `MENGE` (PO Qty), and `NETPR` (Net Price).

### Standard Export Columns (Consolidated)
`PurchaseOrderNumber`, `SupplierCode`, `SupplierName`, `Werkcode` (Plant), `Materialnummer` (Material), `Menge` (Qty), `Einheit` (UOM), `Buchungsdatum` (Posting Date), `Kostenstelle` (Cost Center), `Currency`, `DocumentDate`, `TaxCode`, `MaterialGroup`, `PurchasingOrganization`.

## 2. Common Onboarding Pain Points

1.  **German/English Hybrid Schemas**: SAP's core legacy tables are German-named (`Werk`, `Lieferant`, `Menge`). Exports generated by global instances frequently feature localized headers (e.g., `Buchungsdatum` instead of `Posting Date`), requiring robust translation dictionaries.
2.  **Supplier-Specific Material Codes**: Enterprises lack unified material naming. A single diesel fuel asset might appear as `DSL` (Chevron invoice), `DIESEL-1` (Shell delivery), or `DFUEL` (BP contract) in different corporate regions.
3.  **Inconsistent Units of Measure (UOM)**: Bulk liquid fuel is purchased globally in US Gallons (`gln` or `gal`), Liters (`L`), and cubic meters (`m3`). Gaseous fuels are standard in cubic meters (`m3` / `cbm`) or Energy units (MMBtu), which must be resolved before calculation.
4.  **Mixed Date Formats**: Local SAP users enter dates based on regional settings. A single CSV file can contain a mix of German dot notation (`DD.MM.YYYY`), standard ISO (`YYYY-MM-DD`), and US slashes (`MM/DD/YYYY`).

## 3. Rationale for Problematic Sample Rows (`sap_export.csv`)

Our realistic `sap_export.csv` contains specific operational anomalies to validate parser resilience and analyst workflows:

*   **Row 1 & 2 (Duplicate PO Line - `4500019201`)**: 
    *   *Why it exists:* Multiple identical rows occur in SAP ledgers when a single purchase order undergoes split-shipment deliveries or partial invoice receipts.
    *   *System validation:* Ingested to show calculations but flagged/viewed by analysts to prevent double-counting of carbon scope.
*   **Row 3 (Same Material with different UOM - `gal` vs `L`)**:
    *   *Why it exists:* Chevron billed a US branch in gallons while Shell billed a German branch in Liters for the same material number `DIESEL-1`.
    *   *System validation:* Normalization maps `gal` to standard Liters (`L`) by multiplying by `3.78541`, demonstrating unit standardization.
*   **Row 4 (Missing Supplier Code but Name present, and supplier-specific code `DSL`)**:
    *   *Why it exists:* A manual invoice journal entry in FI bypassed MM procurement master records, resulting in an unmapped supplier code. The buyer manually entered `"DSL"` instead of selecting the standard material code.
    *   *System validation:* The date is successfully parsed (in ISO format). The fuel `DSL` fails to match the regular combustion codes and is flagged as `SUSPICIOUS` (Unrecognized fuel type), requiring analyst review.
*   **Row 5 (Missing Plant Code, supplier-specific `DFUEL`, and hyphenated date `18-05-2026`)**:
    *   *Why it exists:* Corporate cost allocation log. The plant field was left blank because the expense was allocated to a corporate-level Cost Center `CC-US-05` rather than a specific physical plant.
    *   *System validation:* Test cases verify that the parser successfully handles the date (`18-05-2026`) and flags the record due to the unmapped fuel code `DFUEL`.
*   **Row 6 (Negative Quantity `-25 L` and Supplier-specific `PETROL_PREM`)**:
    *   *Why it exists:* Represents a commercial refund / credit memo from ExxonMobil returning unused test fuel.
    *   *System validation:* Triggers the negative value check in the validation engine, flagging the row as suspicious since negative inventory changes must be audit-checked to prevent scope greenwashing.
*   **Row 7 (Invalid Plant Code `9999`)**:
    *   *Why it exists:* Plant `9999` is a dummy testing entity used during ERP testing that was accidentally left active in a production ledger export.
    *   *System validation:* Ingests the natural gas data successfully, but flags it because the plant `9999` is not mapped in the Master Organization site registry.

## 4. Real-World Limitations

*   **Ledge Allocation Ambiguity**: SAP procurement ledgers show *purchases*, not actual *combustion*. A sustainability auditor must verify if purchased fuel was consumed within the audit period or held in stock, which requires cross-referencing inventory table `MARD` (Material Stocks).
*   **Physical Plant Mappings**: Static CSV uploads map Plant Codes (`1100`) to geographic regions. In production, this requires integration with SAP plant master lookup table `T001W` to dynamically assign regional carbon grid intensity factors.

---

## 5. Technical Assumptions

*   **Fuel Densities**: Default density conversion factors are standard EPA/DEFRA values.
*   **Standard Mapping Rules**: If fuel names contain `'diesel'`, they map to Diesel (`2.68 kg CO2e/L`); if they contain `'petrol'` or `'gasoline'`, they map to Petrol (`2.31 kg CO2e/L`). Unrecognized codes fall back to `default_fuel` (`2.50 kg CO2e/L`) and trigger alerts.
